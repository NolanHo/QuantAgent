# 09. 评分体系与高价值事件判定 PRD

## 文档状态

**版本**：v1.1
**状态**：PRD 收口稿
**关联 issue**：#128
**适用范围**：事件筛选、重点事件入选、事件详情解释、审批排序和用户可见评分语义
**不包含**：最终算法权重、训练方案、收益回测、个性化推荐、自动执行放行分

## 为什么需要多层评分

QuantAgent 不能用一个“AI 总分”解释所有判断。操盘者至少需要区分：

- 这条消息来源是否权威。
- 这件事本身是否可信。
- 如果属实，它对行业和标的的影响是否强。
- 现在是否仍处于处理窗口。
- 系统分析结论是否稳定。
- 某条建议是否值得进入人工确认。

这些判断的输入、用途和风险边界不同，必须分层展示。

## 用户可见评分总览

| 名称 | 回答的问题 | 主要使用位置 | V1 展示形态 |
| --- | --- | --- | --- |
| 来源权威度 | 来源本身靠不靠谱 | Dashboard、Events、Event Detail、Approval Detail | 等级 + 简短来源说明 |
| 事件可信度 | 这件事现在有多可能为真 | Dashboard、Events、Event Detail、Approval Detail | 0-100 + 高/中/低 + 验证状态 |
| 行业影响强度 | 属实后对行业和标的冲击多大 | Dashboard、Events、Event Detail | 0-100 + 弱/中/强/极强 |
| 时效性 | 现在是否还值得优先处理 | Dashboard、Events、Approval | 高/中/低 + 时间窗口 |
| 事件综合优先级 | 今天该优先看哪几条事件 | Dashboard、Events | 0-100 + S/A/B/C + 入选原因 |
| 分析置信度 | 当前行业分析结论是否稳定 | Event Detail、Approval Detail | 0-100 + 不确定性摘要 |
| 建议推荐度 | 建议是否值得进入人工确认 | Event Detail、Approvals | 0-100 + 推荐原因 |
| 风险方向 | 建议会增加、降低还是不改变风险 | Approvals、Approval Detail、Approval Link | increase_risk / reduce_risk / neutral |
| 确认等级 | 人类确认需要多强 | Approval Detail、Approval Link | informational / soft_confirm / strong_confirm / link_confirm / manual_only |

## 评分不是执行权限

高分不表示可以自动执行。所有真实或虚盘动作仍必须经过 Decision / Policy Gate。

```text
Event / Analysis scores
  -> DecisionResult
  -> ActionRequest
  -> ApprovalPolicyResolver
  -> ApprovalRequest / Notification / dry_run / reject
```

前端不能把 `建议推荐度`、`分析置信度` 或 `事件综合优先级` 展示成“执行通过率”。

## 评分分层定义

### 来源权威度

回答：`这条消息来自的渠道，本身靠不靠谱？`

典型输入：

- 来源类型：官方公告、监管文件、主流媒体、行业媒体、社交账号、论文、论坛。
- 来源历史可靠性。
- 是否属于用户或行业包配置的核心来源。
- 是否存在历史误报或夸大记录。

展示：

```text
来源权威度：A
说明：主流媒体首发
```

V1 约束：

- 不把来源权威度等同于事件可信度。
- 社交账号或匿名论坛即使时效高，也不能直接形成高可信事件。

### 事件可信度

回答：`这件事本身现在有多可能为真？`

典型输入：

- 来源权威度。
- 是否存在第二独立信源。
- 是否有官方确认。
- 是否存在互相冲突的信息。
- 是否完成搜索或历史上下文校验。

展示：

```text
事件可信度：82 / 100
等级：高可信
验证状态：双信源交叉验证
```

V1 约束：

- 高风险二阶影响至少需要多信源支持；验证失败时必须降低可信度或进入人工确认 / 仅通知。
- 事件存在重大冲突信息时，必须展示冲突摘要，而不是只显示一个分数。

### 行业影响强度

回答：`如果事件属实，它对行业和标的冲击多大？`

典型输入：

- 是否影响核心供需、政策、监管、技术瓶颈或成本结构。
- 影响是否直达市场对象，还是二阶推导。
- 是否影响多个行业或多个标的。
- 行业包给出的 scoring hints。

展示：

```text
行业影响强度：89 / 100
等级：强
说明：直接影响原油供给预期，并影响近月期限结构
```

V1 约束：

- 影响强不代表可信度高。
- 多行业冲突时，详情页应展示主要共识和关键分歧。

### 时效性

回答：`这条事件是不是现在就该看？`

典型输入：

- 发布时间距当前时间。
- 市场是否仍处于可反应窗口。
- 是否有审批到期时间。
- 是否已被后续事件覆盖或澄清。

展示：

```text
时效性：高
窗口：发布后 15 分钟内
```

V1 约束：

- 时效性只能解释优先处理窗口，不解释事件真假。
- 过期或已澄清事件不得继续占用 Dashboard 首屏重点位。

### 事件综合优先级

回答：`在当前时间窗内，操盘者该优先看哪几条？`

V1 必须综合：

- 事件可信度。
- 行业影响强度。
- 时效性。
- 来源权威度。

可选加入：

- 是否已经完成 AI 分析。
- 是否有待审批建议。
- 是否存在系统错误影响该事件分析质量。

展示：

```text
事件优先级：88 / 100
等级：S
入选原因：高可信 + 强影响 + 高时效
```

V1 约束：

- Dashboard 默认展示最高优先级的少量重点事件。
- Events 页面允许在“最新优先”和“高价值优先”之间切换，但默认应支持最新 + 高分混合。

### 分析置信度

回答：`系统对当前行业影响分析本身有多确定？`

典型输入：

- 行业包分析是否完成。
- 关键工具调用是否成功。
- 证据质量和数量。
- 支持观点与反方观点是否冲突。
- 是否存在数据缺口或工具失败。

展示：

```text
分析置信度：74 / 100
不确定性：缺少官方二次确认；存在市场提前计价风险
```

V1 约束：

- 不展示完整模型推理链。
- 必须展示结构化支持观点、反方观点和不确定性摘要。

### 建议推荐度

回答：`这条建议是否值得进入人工确认或优先处理？`

典型输入：

- 事件综合优先级。
- 分析置信度。
- 风险等级和风险方向。
- 触发信息质量。
- 是否满足审批前置条件。

展示：

```text
建议推荐度：78 / 100
推荐原因：高价值事件 + 分析方向清晰，但仍需人工确认
```

V1 约束：

- 它不是胜率、收益预期或执行放行分。
- 审批工作台可以按建议推荐度排序，但必须同时展示风险方向、确认等级和过期策略。

## 异常和降级展示

| 场景 | 展示要求 | 后续处理 |
| --- | --- | --- |
| 单一低权威来源 | 标记来源弱、可信度低或待验证 | 不进入高置信建议 |
| 多信源冲突 | 展示冲突摘要 | 降低可信度，进入人工确认或仅通知 |
| 工具调用失败 | 在分析置信度中展示数据缺口 | 支持重分析 |
| 事件已过期 | 降低时效性，移出重点事件区 | 保留审计和详情 |
| 分析输出无效 | 展示分析失败，不展示建议推荐度 | 允许重试或人工标注 |
| Policy Gate 阻断 | 展示阻断原因和 request_id / trace_id | 不允许前端绕过 |

## 页面使用规则

### Dashboard

重点事件卡只展示少量核心评分：

- 事件综合优先级。
- 事件可信度。
- 行业影响强度。
- 时效性。
- 入选原因。

### Events

事件列表展示：

- 事件综合优先级。
- 事件可信度。
- 行业影响强度。
- 分析状态。
- 涉及行业。

### Event Detail

详情页展示：

- 来源权威度。
- 事件可信度。
- 行业影响强度。
- 分析置信度。
- 建议推荐度。
- 风险点、支持观点、反方观点。

### Approvals

审批页展示：

- 建议推荐度。
- 风险方向。
- 风险等级。
- 确认等级。
- `expires_at` 和 `expiration_action`。
- 事件可信度和分析置信度摘要。

## 非目标

- 不定义最终算法权重。
- 不做收益回测驱动评分。
- 不做个性化学习排序。
- 不做自动执行放行分。
- 不把所有内部评分细节暴露给用户。

## 接入顺序与阶段约束

### 首页基础 UI 先于评分体系接入

Dashboard / 首页第一版先以普通事件展示和 mock UI 跑通结构，不把评分体系作为当前首页基础骨架的前置依赖。

这意味着：

- 首页第一版可以先没有正式评分字段。
- 首页第一版不要求高价值事件排序已经接入。
- 当前首页 mock 中出现的类似评分标签，只服务于基础展示，不代表正式 contract。

### 首个正式接入页面：Events

评分体系第一次正式进入页面能力时，应优先落在 `/events` 高价值事件中心，而不是先改 Dashboard 或事件详情页。

原因：

- `/events` 同时承接重点事件区、事件列表、筛选与排序，更适合作为评分字段第一次正式消费的位置。
- Dashboard 更适合在后续阶段只消费少量重点事件摘要，而不是先承担完整评分接入。
- 事件详情页更适合解释单条事件为什么得到这些分，而不是先承担列表级排序能力。

### 页面接入顺序建议

建议的后续接入顺序：

1. `/events`：先接入事件综合优先级、事件可信度、行业影响强度、时效性和入选原因。
2. Dashboard：后续只补重点事件摘要展示，不回退为评分首次落地点。
3. `/events/:eventId`：补评分解释、不确定性摘要和建议推荐度语义。
4. `/approvals` / `/approvals/:approvalId`：补推荐度、风险方向、确认等级和可信度 / 分析置信度摘要。

### Mock 与正式字段的过渡约束

首页基础骨架阶段可能仍使用临时字段或标签，例如优先级、参考强度、影响方向之类的 mock 表达。这些字段不能长期保留为正式命名。

后续进入 API contract、`packages/contracts`、mock data 和前端字段命名时，应逐步收口到本 PRD 所定义的正式语义，例如：

- `source_authority`
- `event_reliability`
- `impact_strength`
- `freshness`
- `event_priority`
- `analysis_confidence`
- `recommendation_score`
- `verification_status`
- `uncertainty_summary`
