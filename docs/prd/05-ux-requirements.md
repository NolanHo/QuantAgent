# 05. UX 与界面需求

## 设计目标

前端需要服务于高信息密度的量化决策场景。用户应能够快速理解事件来源、系统推理链路、置信度、风险和可执行动作。

设计风格要求为 Apple-quality 工业设计、玻璃拟态背景和高信息密度，但视觉效果不能牺牲信息可读性和交易确认安全性。

## 核心界面

### 思维链视图

以流程图形式实时动态展示 Agent 从观察事件到得出结论的关键步骤。

需要展示：

- 原始事件来源。
- 实体提取结果。
- 行业包路由结果。
- 一级影响。
- 二阶影响。
- 多空辩论摘要。
- 独立信源验证结果。
- 置信度评分。
- 最终建议动作。

交互要求：

- 支持展开查看每一步的证据摘要。
- 支持标记某一步为有问题，作为用户反馈。
- 支持查看当前步骤状态，包括进行中、成功、失败和等待人工确认。

### 行业包管理看板

支持可视化编辑每个行业包的监控目标和关键词。

需要展示：

- Package 名称和启停状态。
- 监控数据源列表。
- 关键词和实体规则。
- Ticker List。
- 最近触发事件。
- 最近失败记录。

操作要求：

- 支持启用或暂停单个行业包。
- 支持新增、编辑和删除监控目标。
- 支持调整关键词过滤规则。
- 支持查看插件健康状态。

### 交易建议弹窗

交易建议弹窗用于承接 HITL 流程。弹窗必须解释为什么建议买入、卖出、做空或观望。

需要展示：

- 建议动作。
- 标的和工具类型。
- 置信度。
- 关键证据。
- 主要反方观点。
- 风险提示。
- 是否允许自动执行。
- 用户操作按钮。

操作要求：

- 置信度 70%-90% 时，用户可确认、拒绝或要求重新分析。
- 无 API 权限市场默认不能出现直接下单动作，只能通知或记录。
- 高风险动作需要二次确认。

## 实时事件流

事件流用于展示系统正在处理的事件和历史事件。

需要展示：

- 事件标题或摘要。
- 来源和时间。
- 当前状态。
- 涉及行业包。
- 置信度。
- 推荐动作。
- 是否需要人工处理。

## 状态设计

| 状态 | UI 表现 |
| --- | --- |
| 加载中 | 展示当前处理步骤，不只显示全局 spinner |
| 空态 | 说明尚无事件、尚未配置行业包或当前过滤条件无结果 |
| 错误态 | 展示失败步骤、原因摘要和可重试动作 |
| 等待确认 | 高亮待用户处理的建议 |
| 已执行 | 展示执行结果和审计记录入口 |
| 已通知 | 展示通知渠道和送达状态 |

## 安全交互要求

- 自动执行开关必须清晰可见。
- 交易确认按钮不得与普通关闭按钮视觉混淆。
- 用户拒绝建议后，系统不得继续自动执行同一建议。
- 所有交易相关操作需要可追溯。

## 可用性要求

- 事件流和思维链视图需要支持长内容折叠。
- 高频事件场景下，需要避免 UI 被持续刷新打断用户阅读。
- 用户应能筛选行业包、状态、置信度和是否待人工确认。
- 关键交易建议应有明显优先级，而不是被普通通知淹没。
