# 05. UX 与界面需求

## 设计目标

前端需要服务于高信息密度的运行时管理场景。用户应能够快速理解事件来源、系统推理链路、插件状态、审批状态和可执行动作。

## 核心界面

### 事件流与详情

用于展示外部事件、当前状态、涉及行业包和审计入口。

需要展示：

- 事件标题或摘要。
- 来源、时间和状态。
- 路由到的行业包。
- 置信度和建议动作。
- 相关审批、通知和执行记录。

### 运行时间线

用于展示 Agent run 和 tool invocation 的结构化过程，不展示完整原始推理链。

需要展示：

- AgentDefinition。
- 使用的 Skill 列表。
- 工具调用步骤和耗时。
- 错误摘要和输出摘要。
- 事件 ID 和 trace ID。

### 审批工作台

用于处理待授权、即将过期、已处理和已过期的请求。

需要展示：

- 建议动作和风险方向。
- expires_at 倒计时。
- expiration_action。
- 关键证据和反方观点摘要。
- approve / reject / request_reanalysis / amend。

### 插件后台

用于展示插件列表、类型、版本、来源、状态和配置。

需要展示：

- 插件启停状态。
- 插件配置表单。
- 插件依赖和错误。
- 插件审计记录。

### Skills / Tools / Industries

用于展示 Skill Registry、Tool Registry、行业包和市场映射摘要。

需要展示：

- Skill 来源、版本和授权状态。
- Tool 来源插件、schema 和权限状态。
- 行业包、Source Binding 和工具摘要。

## 状态设计

| 状态 | UI 表现 |
| --- | --- |
| 加载中 | 展示当前处理步骤，而不是只显示全局 spinner |
| 空态 | 说明尚无事件、尚未配置行业包或当前过滤条件无结果 |
| 错误态 | 展示失败步骤、原因摘要和重试入口 |
| 待确认 | 高亮待处理的审批请求 |
| 已执行 | 展示执行结果和审计入口 |
| 已通知 | 展示通知渠道和送达状态 |

## 安全交互要求

- 交易确认按钮不能与普通关闭按钮混淆。
- 自动执行开关必须清晰可见。
- 用户拒绝后，同一建议不得继续自动执行。
- 敏感字段、密钥和私有策略必须脱敏展示。

## 交互原则

- REST 负责快照，WebSocket 只负责增量提醒。
- 插件配置只支持 schema-driven form。
- 前端只展示结构化运行过程，不展示完整 chain-of-thought。
- 高频事件场景下要避免持续刷新打断阅读。
