# 00. Dashboard

## 页面定位

Dashboard 是操盘者登录后的第一判断面板。它不是传统后台仪表盘，也不是插件治理首页，而是围绕“今天先看什么、哪些建议要处理、系统是否影响判断质量”组织的总控页。

根据 issue #127 评论，V1 保留独立 Dashboard，不让 `/events` 同时承担总首页和事件中心。

## 用户任务

- 快速看到今日最值得关注的事件。
- 快速看到待处理审批和即将过期审批。
- 发现会影响决策质量的关键系统问题。
- 进入事件中心、事件详情、审批工作台或 Runtime 排障页。

## 主对象和真源

Dashboard 聚合多个资源，但主焦点是高价值事件。

| 信息 | 真源 |
| --- | --- |
| 重点事件 | REST 事件快照和评分摘要 |
| 待审批摘要 | ApprovalRequest 查询 |
| 系统健康提醒 | Runtime health / runtime error 摘要 |
| 实时刷新 | WebSocket 只触发刷新，不作为状态真源 |

## 页面结构

```text
页面头
  -> 今日重点事件
  -> 待处理审批
  -> 关键健康提醒
  -> 工作入口
```

首屏优先级：

1. 今日重点事件。
2. 待处理和即将过期审批。
3. 关键健康提醒。
4. 次级工作入口。

## 关键模块

### 今日重点事件

必须展示：

- 默认少量重点事件，具体数量可配置，首版建议 5 条。
- 事件综合优先级。
- 事件可信度。
- 行业影响强度。
- 时效性。
- 入选原因。
- 分析状态。

点击事件进入 `/events/:eventId`。

### 待处理审批

必须展示：

- 待处理数量。
- 即将过期数量。
- 最高优先级的 2 到 3 条审批摘要。
- 风险方向和到期时间。

只提供进入审批工作台或审批详情的入口，不在 Dashboard 内联 approve / reject。

### 关键健康提醒

只展示会影响当前判断质量的问题：

- 关键 source 插件故障。
- 事件分析连续失败。
- 影响当前页面的实时连接降级。
- Tool / Agent 失败影响某条重点事件。

不展示完整运行大盘，也不替代 Runtime 页面。

## 状态与失败路径

| 状态 | 页面行为 |
| --- | --- |
| 初始加载 | 分区 skeleton，不用全页 spinner |
| 无重点事件 | 显示“当前没有高价值事件”，仍保留进入事件中心入口 |
| 无审批 | 审批区显示当前无待处理项 |
| Runtime 异常 | 健康提醒区高亮，并提供进入 Runtime 的入口 |
| 部分资源失败 | 只降级对应模块，保留其他模块 |
| 实时断连 | 展示轻量提醒，并允许 REST 刷新 |

## 风控边界

- Dashboard 不承接任何高风险动作。
- Dashboard 不展示完整模型推理链。
- Dashboard 不展示 secret、私有策略或交易账户敏感信息。
- 任何审批动作必须进入 Approvals 或 Approval Detail。

## 验收口径

必须成立：

- 用户打开首页后能先看到今日重点事件，而不是插件、工具或运行时列表。
- 用户能明确知道是否有待处理或即将过期审批。
- 用户能发现影响当前事件判断质量的关键系统问题。
- Dashboard 不提供绕过审批链路的动作。

失败信号：

- 首页退化成系统资源目录。
- 插件、Skill、Tool、Industry 等治理对象抢占首屏。
- 用户无法从 Dashboard 自然进入事件详情和审批工作台。

## 非目标

- 不做完整事件列表。
- 不做插件治理首页。
- 不做审批内联操作。
- 不做执行器或策略开关页面。
