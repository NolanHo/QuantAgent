# 02. 高价值事件中心

## 页面定位

高价值事件中心是从 Dashboard 进入后的事件浏览和筛选页。它的目标不是做新闻全文阅读器，而是让操盘者在重点事件之外浏览更多事件，并稳定进入事件详情 / 决策页。

## 用户任务

- 查看当前时间窗内的重点事件和全量事件。
- 按时间、行业、可信度、分析状态收窄事件。
- 理解一条事件为什么被判定为高价值。
- 进入事件详情查看行业影响和最佳动作。

## 主对象和真源

主对象是 Event。

| 信息 | 真源 |
| --- | --- |
| 事件列表 | REST `/events` 类资源 |
| 事件状态 | Event 当前状态和 state transitions 摘要 |
| 评分摘要 | ScoredAnalysis / Decision 输入摘要，字段以后续 contract 为准 |
| 实时变化 | WebSocket topic 触发 query invalidation |

## 页面结构

```text
页面头
  -> 事件概览
  -> 今日重点事件区
  -> 筛选与排序
  -> 事件列表
  -> 轻量系统提醒
```

## 关键模块

### 事件概览

建议展示：

- 新事件数。
- 重点事件数。
- 分析中事件数。
- 分析失败或待复核事件数。

### 今日重点事件区

每条重点事件必须展示：

- 标题或摘要。
- 来源和发布时间。
- 事件综合优先级。
- 事件可信度。
- 行业影响强度。
- 时效性。
- 涉及行业。
- 分析状态。
- 入选原因。

重点事件只提供“查看分析”入口，不提供审批动作。

### 筛选与排序

首版筛选：

- 时间范围。
- 行业。
- 事件可信度。
- 分析状态。
- 来源类型。

首版排序：

- 最新 + 高价值混合。
- 最新优先。
- 高价值优先。

筛选和排序进入 URL search params，支持刷新恢复。

### 事件列表

列表行必须让用户能做初筛：

- 事件摘要。
- 来源和时间。
- 当前状态。
- 评分摘要。
- 涉及行业。
- 分析状态。

点击整行进入 `/events/:eventId`。

## 状态与失败路径

| 状态 | 页面行为 |
| --- | --- |
| 初始加载 | 先展示框架和分区 skeleton |
| 无重点事件 | 重点区展示空态，列表仍可用 |
| 无列表结果 | 提示筛选条件无结果，允许清空筛选 |
| 分析中 | 行内展示分析中，不阻塞查看事件事实 |
| 分析失败 | 行内展示失败摘要和进入详情入口 |
| 实时断连 | 顶部轻量提醒，不阻断浏览 |
| API 失败 | 模块级错误和重试入口 |

## 风控边界

- 事件中心不直接 approve / reject / amend。
- 事件中心不展示完整模型推理链。
- 评分必须区分事件可信、事件重要和分析置信度。
- 对低可信或冲突事件必须有可见标签，不能只靠颜色暗示。

## 验收口径

必须成立：

- 用户能区分重点事件和普通事件。
- 用户能知道某条事件为什么值得优先查看。
- 用户能从事件中心稳定进入事件详情。
- 筛选和排序可恢复。

失败信号：

- 页面退化成按时间倒序新闻流。
- 评分、可信度、行业影响混成一个字段。
- 事件中心承担审批或执行动作。

## 非目标

- 不做新闻全文阅读器。
- 不做审批工作台。
- 不做多候选动作比较。
- 不做运行时调试页。
