# 07. 验收标准与待确认项

## 信息完整度评估

| 维度 | 状态 | 说明 |
| --- | --- | --- |
| 目标或业务结果 | 部分明确 | 明确要做事件驱动量化 Agent，但缺少量化指标 |
| 目标用户 | 明确 | 面向专业交易者、研究员和策略开发者 |
| 范围内行为 | 明确 | 包含事件监控、路由、行业分析、审批和受控执行 |
| 范围外边界 | 部分明确 | 已明确不以通用聊天机器人或完整交易终端为目标 |
| 成功标准 | 部分明确 | 已有跨模块验收标准，但缺少部分量化门槛 |
| 时间与合规约束 | 缺失 | 尚未定义发布时间、上线门禁和交易合规边界 |

## 跨模块验收标准

### 产品验收

- 用户能够配置至少一个行业包并看到其监控目标。
- 用户能够看到事件从捕获、路由、分析、验证到建议的完整状态。
- 用户能够理解每条建议的证据、反方观点、风险和置信度。
- 用户能够确认、拒绝或要求重新分析 HITL 建议。

### 行业包验收

- 石油行业包能够处理典型制裁或供应冲击事件。
- 半导体 / 内存行业包能够处理典型技术冲击事件。
- 每个行业包都需要包含 Source Binding、AgentDefinition、Skill 和 Tool。
- 每个行业包都需要通过 Registry 可发现，并能展示健康状态。

### Agent 流程验收

- Router Agent 能从非结构化文本中提取关键实体。
- Router Agent 能给出路由理由。
- Scoring / Debate 至少输出支持观点、反对观点和不确定性。
- 系统能根据 Decision / Policy Gate 进入通知、审批、dry-run 或受控执行。

### 风控验收

- 高风险结论必须完成多信源验证。
- 验证失败时不得自动执行。
- 单一来源高频噪音必须触发熔断或降级。
- 无 API 自动交易权限时不得出现直接执行动作。
- 敏感配置不得出现在前端、日志或仓库文件中。

### UX 验收

- 运行时间线能展示关键步骤和状态。
- 行业包管理看板能展示状态、来源和最近错误。
- 审批工作台能展示建议动作、证据、反方观点和风险。
- 错误态需要说明失败阶段并提供重试或反馈路径。

### WebSocket 与运行时验收

- WebSocket 断线后能恢复当前页面状态。
- REST 快照始终可作为状态恢复基准。
- 插件配置更新后能正确刷新详情与审计记录。
- runtime 目录和 Docker 入口满足部署与重建要求。

## 测试重点

- 典型石油事件 happy path。
- 典型半导体 / 内存事件 happy path。
- 一个事件触发多个行业包的跨行业场景。
- 单一来源高频发文触发噪音熔断。
- 验证失败导致降级为人工确认或仅通知。
- 审批超时和一次性授权页恢复场景。
- WebSocket 断线重连后的状态恢复。
- 插件配置 schema 校验失败和 reload 场景。

## 顶层风险

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 自动执行边界不清 | 可能导致不符合权限或风控要求的执行 | 先固化市场、账户、权限和审批规则 |
| 二阶推理难验证 | 可能产生高置信度但错误的建议 | 强制多信源验证，并保留反方观点 |
| 行业包知识过粗 | 容易把复杂行业关系简化为错误因果链 | 为每个行业包维护版本化知识和样例集 |
| 运行时状态分散 | 可能导致前端与后台状态不一致 | 以 REST 快照、审计和 WebSocket 增量共同收敛 |
| 数据源稳定性不足 | 可能影响事件及时性和可信度 | 建立健康检查、重试和来源置信度 |

## 待确认问题

1. 首批真实接入的交易市场和金融工具范围是什么。
2. 自动执行是否允许，若允许需要哪些账户级风控阈值。
3. 首批通知渠道是否仅限于 UI，还是必须包含外部消息通道。
4. 第三方和私有插件是否允许在 v1 直接进入运行时目录。
5. 是否需要把部分审批动作开放给本地 CLI 或仅限 Web。

## 是否建议进入实施

建议进入原型实施和技术验证，但不建议直接进入真实自动交易上线。

理由：

- 核心流程、插件治理和审批路径已经明确，足以支持原型。
- 交易权限、账户风控和合规边界仍不完整。
- 首阶段应优先实现通知和人工确认模式，自动执行能力应单独评审。
