# 06. 风控、约束与异常处理

## 模块目标

本模块定义系统在推理、数据源、隐私、审批和执行上的约束。由于输出可能影响真实交易，风控优先于自动化效率。

## 幻觉校验

系统对高风险结论必须做多信源交叉验证。

### 要求

- 不允许基于单一社交媒体内容直接形成高置信度建议。
- 高风险二阶影响至少要有两个独立来源支持。
- 验证失败时应降低置信度或进入仅通知 / 人工确认模式。
- 验证过程需要记录数据源、时间和摘要。

## 隐私保护

敏感配置应以 secret reference 为主，必要时才加密入库。

### 敏感信息范围

- API Keys。
- 交易账户信息。
- 策略参数。
- 私有关键词和行业规则。
- 自动执行权限配置。

### 要求

- 敏感配置不得提交到代码仓库。
- 前端不得暴露不必要的敏感信息。
- 日志和 API 响应不得输出完整密钥或私有策略。

## 噪音熔断

当单一来源短时间内产生大量噪音时，系统应自动暂停该源的高成本路由。

### 要求

- 支持按账号、域名、数据源插件或来源维度统计频率。
- 熔断后保留原始记录，但不继续进入高成本推理链路。
- 人工解除熔断时需要记录操作者和原因。

## 执行风险约束

### 自动执行前置条件

系统只有在同时满足以下条件时才允许进入受控执行：

- 置信度足够高。
- 市场和账户具备自动执行权限。
- Decision / Policy Gate 明确放行。
- 完成独立信源验证。
- 未触发噪音熔断、数据异常或权限限制。

### 禁止自动执行场景

- 无 API 自动交易权限。
- 验证失败。
- 置信度不足。
- 事件存在重大冲突信息。
- 用户或系统风控开关关闭。

## 运行时与插件约束

- 插件不能直接持有数据库 session。
- 插件不能自己启动后台轮询线程或绕过 Scheduler。
- Tool 和 Skill 只能通过 Registry 授权后注入 Agent。
- 热重载只保证开发和后台管理场景可用，不承诺生产零中断。

## 数据源异常

### 异常类型

- API 请求失败。
- 来源被限流或拒绝访问。
- 数据延迟或重复。
- 原文内容被删除或修改。
- 多信源结论冲突。

### 处理要求

- 记录失败原因和重试次数。
- 不因单一数据源失败中断整个系统。
- 关键数据源失败需要 UI 提示。
- 冲突时降低置信度并进入人工确认或仅通知模式。

## 审计要求

所有可能影响建议或执行的动作都需要可追溯。

至少记录：

- 事件 ID。
- 路由结果。
- 分析摘要。
- 验证来源。
- 置信度。
- 审批结果。
- 执行动作和结果。
- 异常和熔断记录。
